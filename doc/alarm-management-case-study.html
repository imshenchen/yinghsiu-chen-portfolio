01

專案概述OVERVIEW · 平台級告警管理模組

DIAEAP+ 是部署於廠內的 AIoT 機聯網平台。本模組要回答一個問題：當底層服務（PLC → … → RabbitMQ）發生異常時，如何被監控到、被判斷成告警、被送到對的人手上，並在平台前端被妥善管理。本案例聚焦在「如何用開源模組組裝這條鏈，並設計平台端的操作介面」。

◷監控 Observability

以 Odigos 監控從 PLC 往上到 RabbitMQ 之間的所有服務，並建立數據血緣（data lineage），讓異常可被追溯來源。

⚑告警 Alerting

異常觸發後，串接 Keep 進行彙整與發報，初期渠道為 Email 與 EAP+ 平台的「小鈴鐺」通知。

▤平台管理 UI

在 EAP+ 前端提供告警列表、小鈴鐺通知與通知渠道設定；規則設定 UI 列為後續階段。

✓ MVP 範圍內

05

開源整合OPEN SOURCE · 三個模組的職責分工

這個模組的本質是整合而非重造。三個開源元件各司其職，平台的價值在於把它們黏合成一條可被前端操作的告警鏈。

Odigos
監控層 · OBSERVABILITY
  • 監控 PLC → RabbitMQ 之間的所有服務
  • 建立數據血緣，讓告警可回溯來源服務
  • 作為規則引擎的觀測數據基礎
Grafana
規則引擎 · RULE ENGINE
  • 支援 Loki / InfluxDB / Prometheus / Infinity / SIE-edgeX 等資料源
  • 多條件查詢與複雜表達式設定規則
  • Direct HTTP REST API 建立規則判斷
  • 限制：免費版規則上限 100 條
Keep
彙整發報 · AIOps
  • 警報彙總與資料拓樸（correlation）
  • Workflow automation + provider 推送多渠道
  • 提供取得警報列表 API；變更狀態 API 待評估
  • 支援 YAML 與 UI Builder；可串 Incidents 異動與 AI 改善建議

整合介面摘要

規則層以 REST API 對 Grafana 建立 / 查詢規則；發報層以 Keep 的 Workflow（HTTP + YAML）定義 actions，將告警同步推送至 Email、Slack、Webhook 及 EAP+ 小鈴鐺；前端透過 Keep 列表 API 取得事件，並（待評估）回寫告警狀態。LLM 作為橫貫能力，對事件提供研判與處理建議。
