把 AI 接進
機聯網平台

在設備自動化控制系統控制台中，設計並落地一套「AI 服務」模組： 以代理（Agent）為核心、MCP 串接機聯網即時數據、 並以可視化的思考流程協助現場人員排查異常。 本專案說明其架構分層與實現方式，並以真實用戶訪談回扣設計決策。

02

使用者研究USER RESEARCH · 現場深度訪談

AI 服務上線後，我們對實際使用的產線人員進行深度訪談，涵蓋「使用情境、頻率、看法、操作、滿意度」五個面向。 以下是被整理為設計輸入的關鍵發現。

2小時
導入前需配合班次人工記錄一次資料；AI 服務目標是取代這類重複登錄。
~3分鐘
瓶頸工站人員需在系統中手動找資料比對，約耗 2–3 個生產週期。
4/ 4
現階段滿意度評分；唯一扣分點為「AI 尚無法進行深度思考」。
Scenario

①異常驅動的查詢

使用頻率約「一天一次或一週數次」，多在突發異常、找原因時臨時開啟。 例：晚班標準應做 200 ps、實際只有 140 ps，用 AI 判斷是延誤或其他變動。

Value

②縮小排查範圍

AI 分析答案會被當作佐證；當人員找不到原因時，可看出某時段實際 CT 為空， 藉此縮小排查方位，相對可靠。

Gap

③缺「時段歷史」

目前 UPH／瓶頸站多為整線或當下即時數據， 但現場其實還需要「特定時段內」的歷史資訊來追溯問題。

Friction

④取數帶雜訊

介面直觀，但把數據複製出來使用時會夾帶多餘符號。 查詢分析介面不需華麗視覺，數據有用即可。

Mental Model

⑤名詞不對齊

生產週期 CT 有多種稱呼，換個問法回答就不同，但對產線而言指的是同一件事， 需要術語對齊。

Trust

⑥相信系統不相信人

人員更傾向採信系統數據；期待把延誤性問題與設備異常細項整合， 並拉長時間維度（CT、設備狀態、報警 3→5）以看出長期問題。

想查隨時都可查 — 電腦就在產線隔壁，遇到突發異常就臨時開 AI 來找原因。 — 使用情境 · 訪談摘錄
換了不同的問法或名稱，AI 回答的內容就會不一樣，但對產線來說其實都是同一件事。 — 滿意度 · 訪談摘錄

04

功能實現IMPLEMENTATION · 關鍵畫面與互動

以下四個關鍵功能對應架構的四層落地。UI mock 依 Figma 設計重建； 若要替換為實際畫面，可將 Figma 匯出圖放進各區塊的 SCREEN-SLOT 佔位。

介面層 · 代理層

代理選擇與分類

進入 AI Tool 即以「挑選合適代理」為起點。代理依任務分類，支援最愛、單代理與多代理綜合。

  • 分類標籤：智能助手 / 數據圖表 / 設備排查 / 文檔問答
  • 每張代理卡含名稱、描述與能力標籤（如「設備排查」「文檔問答」）
  • 內建基礎代理：PMM 生產／設備管理、EAP+ 操作／教學
  • 「多代理綜合」可組合多個代理協同回答
推理與工具層

可視化的 AI 思考流程

回答前，AI 以可展開的步驟揭露推理過程；檢索失敗時明確標示，避免「只看到思考、卻沒有結果」的黑箱感。

  • 步驟化：搜尋查詢 → 文檔檢索 → 完成
  • 異常 / 失敗狀態以訊號色標示，並回報失敗關鍵字
  • 檢索無果時主動請使用者補充背景再查
  • 每則回答附 讚／倒讚，回收進問答回饋管理
介面層

兩種對話型態：懸浮式與嵌入式

呼應「離開產線時最常使用、且想查隨時可查」的情境，對話介面同時支援嵌入一頁式與懸浮式（可收合 / 展開）。

  • 嵌入一頁式：在分析頁旁直接展開完整對話
  • 懸浮式：收合為入口、展開為浮層，不打斷當前頁面
  • 模型可切換（如 deepseek），對話依代理載入常見問題
  • 聊天紀錄以「今天 / 昨天 / 日期」分組，便於回溯
資料層 · 回饋迴圈

平台側管理：讓 AI 可被治理

AI 服務不只是聊天框，而是一組可被工程師管理的模組，確保資料來源、模型與回饋皆可控可調。

  • 代理程式管理：建立 / 設定各任務代理
  • 模型管理：登錄與切換可用模型
  • MCP 管理：設定工具與機聯網數據連接
  • 資料集：管理可被檢索的數據與文檔
  • 問答回饋管理：彙整 讚／倒讚 作為優化依據

06

研究 → 實現FROM INSIGHT TO FEATURE

每一項訪談洞察，都對應到架構或功能上的一個具體決策。這是本專案最核心的設計敘事。

使用者洞察對應實現
名詞不對齊：CT 有多種稱呼，換問法答案就不同 → 術語對齊層（同義詞對照） 在推理層加入術語映射，將不同稱呼正規化為同一指標。
缺「特定時段內」的歷史資訊，現多為即時 / 整線數據 → 時間區間查詢 資料層開放時段化歷史查詢（CT / UPH / 報警），支援回溯。
把數據複製出來時會夾帶多餘符號 → 純文字 / 結構化複製 輸出層提供乾淨複製，避免格式符號混入後續使用。
AI 尚無法深度思考（唯一扣分點） → 可視化多步推理 + 失敗標示 揭露搜尋→檢索→彙整步驟，檢索失敗即明示並請求補充。
異常驅動、想縮小排查範圍 → 設備排查代理 + 標準/實際比對 比對標準與實際產出，定位異常時段，縮小排查方位。
希望延誤與設備異常細項整合、拉長時間維度 → 多代理綜合 + 長期整合視圖 將 CT、設備狀態、報警（3→5）整合於同一長期視圖。
07

後續規劃ROADMAP · 從訪談導出的開發方向

訪談中尚未被滿足的需求，構成下一階段的開發重點。

NEXT

術語對齊與時段歷史查詢

優先解決名詞不對齊與「特定時段內歷史」缺口——這是現場最直接的痛點。

PHASE 2

延誤性問題的整合分析

把延誤的多個細項與設備異常原因整合在一起，降低逐項排查的負擔。

PHASE 3

長期趨勢整合視圖

跨時間整合 CT、設備狀態與報警變化，讓長期問題更容易被看見。

PHASE 3

深化推理與回饋閉環

以問答回饋（讚／倒讚）持續校正代理與術語，逐步提升「深度思考」體感。
