背景 · Context

當「設計師畫得多快」成了整個團隊的速度上限

在傳統流程裡，每一支功能都得先經過 product designer 收需求、定義範圍、產出設計稿與 PRD， 前後端工程師才能動工。設計師是所有功能的單一入口，也因此成為整條產線的瓶頸。

這個專案的目標，是把設計師的角色從「逐支功能的繪圖者」轉變為「共用基礎的維護者」： 把可重複使用的設計決策一次性沉澱進元件庫與一份給 AI 讀的設計規範， 再把功能規格的撰寫權交回給最了解該功能的開發者，讓 AI 依據共用文件並行產出。

轉型的三個支點

  • 把重複的設計決策抽成元件庫，而非每次重畫
  • 把設計規範寫成 design guide.md，讓 AI 直接讀
  • 把 PRD 拆成每支功能一份 prd.md，由開發者撰寫、AI 共用
傳統工作流程 · Before

瀑布式序列交付：一支接一支，後段一直在等前段

設計師串接需求、設計與 PRD 三個階段，是所有功能唯一的起點。前後端必須等設計完成才能開工，問題往往到 QA 才暴露。

舊流程遇到的開發問題 · Pain Points

瓶頸不只是「慢」，而是六個彼此放大的結構問題

序列式交付在功能變多、團隊變大後，問題會互相加乘——設計師越忙、交接越多、重工越深。

P-01

設計師是單點瓶頸

每支功能都得等設計師畫完才能往下走。設計產能直接等於團隊的開發上限，人一忙，整條線就塞住。

P-02

只能序列開發

功能一支接一支進入產線，無法重疊。即使工程師有空，也只能等前一支的設計與規格就緒。

P-03

交接造成規格失真

設計稿與 PRD 在「設計師 → 前端 → 後端」之間多次轉手，細節與意圖在交接中遺漏，做出來常常不是原意。

P-04

前端 UI 反覆重工

沒有共用元件庫，每支功能的按鈕、表格、表單都重刻一次，樣式不一致，維護成本隨功能數線性累積。

P-05

回饋週期過長

問題往往到 QA 階段才被發現，此時設計與開發都已完成，修改的代價最高、來回最痛。

P-06

文件靜態、與程式碼脫節

PRD 與設計規範散落在投影片與雲端文件，無法被工具直接讀取，上線後也很快過期、沒人維護。

新工作流程 · After

共用基礎 ＋ 各自寫 PRD ＋ AI 並行開發 ＋ 團隊共同 review

設計師把可重複的設計決策一次沉澱成元件庫與 design guide；每位開發者各自釐清功能、寫好 prd.md， 各團隊的 AI 共用這些文件並行開發多支功能，最後團隊一起 review。

新舊流程比較 · Comparison

同一件事，兩種做法

把兩張流程圖的差異攤平成對照表——關鍵不在「用了 AI」，而在於把設計決策與規格變成可重複使用、可被工具讀取的共用資產。

面向傳統流程（Before）新流程（After）
設計師角色逐支功能繪製設計稿與 PRD，是唯一起點維護共用元件庫與 design guide，做基礎而非逐支產出
規格由誰寫設計師統一定義所有功能每位開發者釐清自己的功能、撰寫 prd.md
開發節奏序列：一支接一支，後段排隊等待並行：多支功能同時開發
UI 一致性每支重刻，樣式漂移、重工累積共用元件庫 + 規範，AI 直接套用
文件型態投影片／雲端文件，工具讀不到、易過期.md 文件，AI 可讀、隨程式碼版控
AI 的角色無，或僅零星輔助各團隊共用同一份文件，作為主要產出引擎
品質把關問題到 QA 才暴露，修改代價最高團隊共同 review，合併前把關一致性與規格

關鍵產物 · Key Artifacts

讓新流程跑得動的兩份共用資產

新流程的可行性，建立在兩份「一次建立、全團隊重複使用」的資產上：一個真實的前端元件庫，與一份專為 AI 撰寫的設計規範。以下為實際範例。

Artifact 01 · Components Library

FAS Design System

Delta 共用 React 元件庫，提供設計 Token、元件與 Dark Mode 支援。AI 開發功能時直接 import 既有元件，不再重刻 UI，也讓多支並行開發的功能保有一致的視覺語言。元件以 Storybook 完整展示， 可作為設計師與開發者之間的共同參照。

Artifact 02 · design guide.md

components-usage.md

一份框架無關、專為 AI 撰寫的設計規範，描述「視覺與互動決策」而非實作細節。它把間距系統、頁面模板、 元件使用準則寫成可被 Claude / Cursor / Copilot 直接讀取的規則，讓 AI 在生成程式碼時自動遵循 FAS Design System 的設計語言。以下為其中「間距系統」的實際節錄：

  • 4pxMicro — 圖示與文字間距、徽章偏移等最小內距
  • 8pxTight — 同一群組內的項目（表單區塊內的欄位）
  • 16pxDefault — 不確定時的通用間距
  • 24pxLoose — 同一頁不同群組之間
  • 32pxSection — 獨立內容區塊之間、頁面外距
閱讀 design guide
My Role · 我的角色

我負責主導設計系統定義，建立統一的設計規範與元件庫，並把設計決策抽象成 AI 可讀的 design guide；同時定義新流程中的 PRD 撰寫標準， 使開發者能各自寫出 prd.md、讓 AI 依共用文件直接讀取規格開發，無需設計師全程介入， 大幅縮短開發時程、減少前端重工。

成效 · Outcomes

瓶頸消散後，團隊節奏改變的三件事

▲ 並行產能

多支功能同時開發，整體產能不再被單一設計師的繪圖速度鎖死，而是隨開發人數擴展。

▲ UI 一致性

共用元件庫 + design guide 讓 AI 產出的介面天生一致，前端重工大幅下降，樣式不再漂移。

▲ 文件即真實

PRD 與設計規範以 .md 隨程式碼版控，AI 與人讀的是同一份事實來源，文件不再上線即過期。
